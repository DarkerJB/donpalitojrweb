1. INTRODUCCIÓN
1.1 Propósito
El presente documento constituye la Especificación de Requisitos de Software (ERS) del Sistema de Comercio Electrónico "Don Palito Junior", elaborado bajo el estándar IEEE 830-1998. Su propósito es definir de manera formal y detallada los requisitos funcionales y no funcionales del sistema, sirviendo como base contractual y técnica entre el equipo de desarrollo y los propietarios del negocio.
El sistema comprende cuatro componentes interconectados: la tienda web (React + Vite), la aplicación móvil (React Native + Expo SDK 52), el panel de administración (React + Vite) y el backend compartido (Node.js + Express 5 + MongoDB Atlas). Este documento abarca los tres roles del sistema: Cliente, Administrador y Sistema (procesos automáticos).
Este documento está dirigido a: el equipo de desarrollo del proyecto, los propietarios de la Cafetería Don Palito Junior, y los instructores evaluadores del programa Tecnología en Análisis y Desarrollo de Software del SENA.

1.2 Alcance
El Sistema de Comercio Electrónico Don Palito Junior es una plataforma digital multiplataforma destinada a digitalizar y modernizar las operaciones de la Cafetería Don Palito Junior, ubicada en la Cra 47 #76D Sur-37, Sabaneta, Antioquia.
El sistema permite a los clientes explorar el catálogo de productos, gestionar su carrito de compras, realizar pagos con tarjeta (Stripe) o por transferencia bancaria, recibir facturas en formato PDF, consultar el historial de pedidos y calificar productos. A los administradores les permite gestionar el inventario, procesar pedidos, crear cupones de descuento, visualizar estadísticas de ventas y exportar datos en CSV y PDF.
El sistema no abarca: gestión de empleados ni control de horarios, integración con sistemas de punto de venta físico, ni procesamiento de pagos en efectivo.

1.4 Definiciones, Acrónimos y Abreviaturas

Término	Definición
RF-XX (Requisito Funcional)	Lo que el sistema debe hacer para satisfacer las necesidades del usuario. Son las características que el usuario detecta directamente.
RNF-XX (Requisito No Funcional)	Restricciones sobre cómo funciona internamente el sistema: rendimiento, seguridad, escalabilidad, etc.
CU-XX (Caso de Uso)	Descripción de la interacción entre un actor y el sistema para lograr un objetivo específico.
HU-XX (Historia de Usuario)	Descripción breve en lenguaje natural de una funcionalidad desde la perspectiva del usuario (Rol: COMO… QUIERO… PARA…).
API (Application Programming Interface)	Conjunto de reglas y protocolos que permite a las aplicaciones comunicarse entre sí para intercambiar datos y funcionalidades.
JWT (JSON Web Token)	Estándar abierto (RFC 7519) para transmitir información de forma segura entre partes como un objeto JSON firmado digitalmente.
Clerk	Servicio de autenticación como plataforma (Auth-as-a-Service) que gestiona el registro, login, OAuth y emisión de JWT en el sistema.
OAuth	Protocolo de autorización estándar que permite el inicio de sesión con cuentas de terceros (Google) sin compartir contraseñas.
Stripe	Plataforma de pagos en línea que procesa transacciones con tarjeta mediante la API PaymentIntent y webhooks HTTPS.
MongoDB Atlas	Servicio cloud de base de datos NoSQL documental. Aloja las 6 colecciones del sistema: Users, Products, Carts, Orders, Coupons, Reviews.
Mongoose	Librería ODM (Object Document Mapper) para Node.js que facilita la interacción con MongoDB mediante esquemas y modelos.
Inngest	Plataforma de background jobs que ejecuta funciones asíncronas en respuesta a eventos, como la sincronización de usuarios Clerk → MongoDB.
Cloudinary	Servicio CDN de gestión y almacenamiento de imágenes en la nube. Almacena y sirve las imágenes de los productos del sistema.
Nodemailer	Librería de Node.js para el envío de correos electrónicos. Gestiona las notificaciones de estado de pedido y el envío de facturas PDF.
PDFKit	Librería de Node.js para generación de documentos PDF. Se usa para generar las facturas de compra del sistema.
Webhook	Mecanismo HTTP mediante el cual un servicio externo notifica al backend de un evento (ej: pago confirmado, usuario creado).
ngrok	Herramienta que crea un túnel HTTPS público hacia el servidor local. Usada para exponer el backend durante el desarrollo y la presentación académica.
CORS (Cross-Origin Resource Sharing)	Mecanismo de seguridad del navegador que controla qué dominios pueden hacer peticiones a la API del backend.
Código QR (Quick Responsive code)	Código de barras bidimensional que en el sistema representa los datos de transferencia bancaria para el método de pago alternativo.
CSV (Comma-separated values)	Formato de archivo de texto plano para representar datos tabulares. Usado para la exportación de datos de ventas e inventario.
WCAG (Web Content Accessibility Guidelines)	Directrices del W3C para mejorar la accesibilidad del contenido web. El sistema sigue pautas básicas de WCAG 2.1.
Carrito híbrido	Implementación del carrito de compras que opera en localStorage para usuarios no autenticados y se sincroniza con el servidor al iniciar sesión.
Estado de pedido	Valor del ciclo de vida de un pedido en el sistema. Los 7 estados posibles son: pending, paid, in_preparation, ready, delivered, canceled, rejected.
Rol de usuario	Atributo que determina los permisos de acceso en el sistema. Los roles son: "user" (cliente) y "admin" (administrador). Asignado y verificado por Clerk.
Panel Admin	Aplicación web independiente (puerto 5174) que provee las herramientas de gestión para usuarios con rol admin.
Tienda Web	Aplicación web principal (puerto 5173) orientada a los clientes para navegación, compra y gestión de perfil.

1.6 Resumen

Este documento especifica los requisitos del Sistema de Comercio Electrónico Don Palito Junior en su versión 3.0. La versión 3.0 actualiza la ERS original (v2.3, febrero 2025) para reflejar el sistema tal como fue construido y entregado: una plataforma multiplataforma con tienda web, aplicación móvil, panel de administración y backend API REST, integrados con servicios cloud de autenticación (Clerk), pagos (Stripe), almacenamiento de imágenes (Cloudinary), background jobs (Inngest) y email transaccional (Nodemailer).

El documento contiene 51 requisitos funcionales (RF-01 a RF-51), 20 requisitos no funcionales (RNF-01 a RNF-20), 51 casos de uso (CU-01 a CU-51), 51 historias de usuario (HU-01 a HU-51), diagramas de actividades para los 51 RF, diagramas de estados para los RF con ciclos de vida (pedidos, cuenta de usuario, cupones) y diagramas de secuencias para los flujos principales del sistema.

 
2. DESCRIPCIÓN GENERAL
2.1 Perspectiva del Producto
El Sistema de Comercio Electrónico Don Palito Junior es una plataforma digital nueva, desarrollada desde cero para la cafetería, que no sustituye ningún sistema previo. Opera como un sistema distribuido con tres clientes independientes que consumen una única API REST compartida.
La tienda web es accesible desde cualquier navegador moderno en dispositivos de escritorio, tablet y móvil. La aplicación móvil está disponible para Android e iOS mediante React Native + Expo. El panel de administración es una aplicación web de acceso restringido, disponible únicamente para usuarios con rol "admin".
El sistema se integra con los siguientes servicios externos: Clerk para autenticación y gestión de identidad, Stripe para procesamiento de pagos con tarjeta, MongoDB Atlas como base de datos cloud, Cloudinary para almacenamiento y servicio de imágenes, Inngest para ejecución de background jobs, y Nodemailer + Gmail para envío de correos electrónicos transaccionales.
El backend es stateless: no almacena estado de sesión. Toda la autenticación se delega a Clerk mediante tokens JWT verificados en cada petición. Esto garantiza escalabilidad horizontal y compatibilidad con despliegues cloud.

2.2 Funcionalidad del Producto
El sistema ofrece las siguientes funcionalidades principales organizadas por componente:

Componente	Funcionalidades Principales
Tienda Web (React + Vite)	Catálogo con filtros y búsqueda. Detalle de producto con imágenes, precio y calificaciones. Carrito híbrido (local/servidor). Checkout en 3 pasos con Stripe y transferencia. Cupones de descuento. Factura PDF por email. Historial de pedidos. Perfil con direcciones y wishlist. Mapa interactivo de la cafetería.
App Móvil (React Native + Expo)	Catálogo, búsqueda y detalle de producto. Carrito sincronizado con servidor. Login con email, contraseña y Google. Historial de pedidos. Perfil extendido con direcciones y wishlist. Sección Visítanos con mapa y horario.
Panel Admin (React + Vite)	Dashboard con estadísticas de ventas y pedidos. CRUD de productos con subida de imágenes a Cloudinary. Gestión de pedidos con cambio de estado. Gestión de clientes (activar/desactivar). CRUD de cupones de descuento. Exportación de datos en CSV y PDF.
Backend API REST (Node.js + Express 5)	API con 40+ endpoints organizados en 9 módulos. Autenticación Clerk JWT. Control de roles (user/admin). Procesamiento de pagos Stripe con webhooks. Sincronización de usuarios con Inngest. Generación de facturas PDF con PDFKit. Envío de emails con Nodemailer. Almacenamiento de imágenes con Cloudinary.

2.3 Características de los Usuarios

Tipo de Usuario	Descripción y Actividades
Cliente (Rol: "user")	Usuario final de la cafetería. Puede ser invitado (no autenticado) o cliente registrado. Navega el catálogo, agrega productos al carrito, realiza compras, aplica cupones, recibe facturas por email, consulta el historial de pedidos y califica los productos que ha comprado. Accede a la tienda web y a la app móvil. Se registra mediante email/contraseña o Google OAuth a través de Clerk.
Administrador (Rol: "admin")	Personal de gestión de la cafetería (propietarios o encargados). Gestiona el inventario de productos, procesa y actualiza el estado de los pedidos, crea y administra cupones de descuento, visualiza estadísticas del negocio y exporta datos. Accede exclusivamente al panel de administración con rol admin asignado en Clerk Dashboard.
Sistema (Actor técnico)	Procesos automáticos del sistema sin intervención humana directa. Incluye: sincronización automática de usuarios Clerk → MongoDB mediante Inngest, envío automático de emails de notificación al cambiar el estado de un pedido, generación automática de facturas PDF al confirmar un pago, y procesamiento automático de pagos mediante webhooks de Stripe.

2.4 Restricciones
•	El sistema no gestiona inventario físico en tiempo real (no hay integración con escáneres de código de barras ni sistemas de punto de venta físico).
•	Los pagos en efectivo no están contemplados en la plataforma digital; el sistema solo soporta pago con tarjeta (Stripe) y transferencia bancaria con QR.
•	La administración de roles de usuario (asignar o revocar el rol "admin") requiere acceso directo al Clerk Dashboard; no se realiza desde el panel de administración.
•	Las categorías de productos están definidas como un enum fijo en el modelo de datos del backend: Palitos Premium, Cocteleros, Dulces, Especiales, Nuevos. Modificar el catálogo de categorías requiere una actualización del esquema del backend.
•	El sistema requiere conexión a internet para operar, ya que depende de servicios cloud externos (Clerk, Stripe, MongoDB Atlas, Cloudinary).
•	Las reseñas de productos son solo calificaciones numéricas (1 a 5 estrellas), sin texto de comentario libre.
•	La aplicación móvil no soporta pagos con tarjeta integrados directamente; el flujo de pago se completa a través de la tienda web o por transferencia.
2.5 Suposiciones y Dependencias
•	Se asume que los propietarios disponen de acceso a internet estable durante la operación del sistema.
•	Se asume que el equipo de administración accederá al panel desde dispositivos con navegador moderno (Chrome, Firefox o Edge versión 2023 o superior).
•	El sistema depende de la disponibilidad de los servicios externos: MongoDB Atlas (SLA 99.95%), Clerk, Stripe y Cloudinary. Una caída de cualquiera de estos servicios afectará la funcionalidad correspondiente.
•	Se asume que los clientes que realizan transferencia bancaria enviarán el comprobante de pago al número de WhatsApp del negocio dentro de las 24 horas posteriores al pedido.
•	La presentación académica del sistema se realiza con el backend corriendo localmente y expuesto mediante un túnel HTTPS de ngrok.
•	Para la publicación de la app móvil en tiendas (Google Play / App Store), se requiere contar con cuentas de desarrollador activas, lo cual no está incluido en el alcance del proyecto académico.
2.6 Evolución Previsible del Sistema
•	Implementación de módulo de gestión de categorías independiente en el panel de administración, sin necesidad de modificar el esquema del backend.
•	Integración de informes de ventas por período con filtros de fecha en el dashboard del administrador.
•	Auditoría formal de accesibilidad WCAG 2.1 con herramientas automatizadas (axe, Lighthouse) y corrección de hallazgos.
•	Implementación de paginación server-side en la API de pedidos para soporte de grandes volúmenes de transacciones.
•	Soporte de pagos móviles integrados en la app (Stripe Mobile SDK) para completar el flujo de compra directamente desde la aplicación.
•	Integración con plataformas de domicilios (Rappi, iFood) mediante API para ampliar el canal de ventas.
•	Sistema de notificaciones push en la app móvil para actualizaciones de estado de pedido en tiempo real.

CU-01: REGISTRO DE CLIENTES.

Código RF	RF-01
Caso de Uso	CU-01: Registro de Clientes
Descripción	Este caso de uso permite a un nuevo usuario crear una cuenta de cliente en Don Palito Junior mediante email/contraseña o Google OAuth, usando Clerk como proveedor de autenticación.
Precondiciones	1.	El usuario no debe estar registrado previamente en el sistema.
2.	El usuario debe tener acceso a internet y un navegador compatible o la app móvil.
3.	El usuario debe disponer de un correo electrónico válido o una cuenta de Google activa.
Secuencia Normal	Pasos - Acciones	Resultados Esperados
	1.	El usuario accede a la tienda web o app y da clic en "Registrarse" o en el ícono de usuario.	El sistema muestra la pantalla de registro de Clerk con opciones: email/contraseña o Continuar con Google.
	2.	El usuario elige el método de registro: email/contraseña o Google OAuth.	Si elige email/contraseña, el sistema muestra campos de nombre, email y contraseña. Si elige Google, se abre el selector de cuenta.
	3.	El usuario completa el formulario y acepta los términos y condiciones.	El sistema valida los campos. Si el email ya existe o los datos no son válidos, muestra mensaje de error.
	4.	El usuario confirma el registro.	Clerk crea la cuenta. Inngest dispara el evento "clerk/user.created" y sincroniza automáticamente el usuario en MongoDB con rol "user".
	5.	El sistema envía un email de bienvenida al usuario.	El usuario recibe el correo (Nodemailer) y queda autenticado en el sistema.
Postcondiciones	1.	El usuario tiene una cuenta activa con rol "user", sincronizada entre Clerk y MongoDB.
2.	Puede iniciar sesión y acceder a todas las funcionalidades del cliente.
Alternativas	1.	Si el correo ya está registrado, Clerk muestra que la cuenta ya existe.
2.	Si la contraseña no cumple los requisitos mínimos, el sistema muestra las validaciones requeridas.
3.	Si el usuario no acepta los términos, el sistema no permite completar el registro.
4.	Si ocurre un error de conexión con Clerk, el sistema muestra un mensaje de error y sugiere intentar nuevamente.

CU-02: INICIO DE SESIÓN DEL CLIENTE.

Código RF	RF-02
Caso de Uso	CU-02: Inicio de Sesión del Cliente
Descripción	Este caso de uso permite a los clientes registrados iniciar sesión en la tienda web o app móvil mediante email/contraseña o Google OAuth.
Precondiciones	1.	El cliente debe estar registrado con una cuenta activa (isActive: true).
2.	El cliente debe conocer sus credenciales o tener acceso a su cuenta de Google.
Secuencia Normal	Pasos - Acciones	Resultados Esperados
	1.	El cliente accede a la tienda web o app y da clic en "Iniciar sesión".	El sistema muestra la pantalla de login de Clerk con opciones: email/contraseña o Continuar con Google.
	2.	El cliente ingresa sus credenciales o selecciona Continuar con Google.	El sistema valida el formato del email. Si es Google OAuth, se abre el selector de cuenta.
	3.	El cliente confirma el inicio de sesión.	Clerk valida las credenciales y emite un JWT con el rol del usuario (publicMetadata.role).

3.	Puede acceder a historial, perfil y proceso de checkout.
Alternativas	1.	Si el correo no está registrado, Clerk muestra un mensaje de error.
2.	Si la contraseña es incorrecta, Clerk permite reintentar y ofrece la opción de recuperar contraseña.
3.	Con múltiples intentos fallidos, Clerk puede bloquear temporalmente el acceso por seguridad.
4.	Si la cuenta está desactivada (isActive: false), el backend retorna 403 y el sistema muestra una página de cuenta inactiva.

3.3 Requisitos Comunes de Interfaces
3.3.1 Interfaces de Usuario
La interfaz de la tienda web está construida con React 18 + Vite 5 y estilizada con Tailwind CSS y DaisyUI. El diseño refleja la identidad visual de la cafetería Don Palito Junior mediante una paleta de colores cálidos, tipografía legible y componentes accesibles.

La interfaz es completamente responsiva, adaptándose a dispositivos móviles, tablets y escritorio. La navegación es intuitiva: menú principal con acceso a catálogo, carrito, perfil e historial de pedidos. Las imágenes de productos son gestionadas mediante Cloudinary con carga optimizada (lazy loading).
El panel de administración es una interfaz separada que ofrece vistas de inventario, pedidos, estadísticas y configuración. Ambas interfaces comparten el mismo sistema de autenticación vía Clerk.
3.3.2 Interfaces de Hardware
El sistema no requiere integración con hardware especializado para su operación principal. Opera sobre infraestructura estándar de navegador web y dispositivos móviles convencionales.
Se contempla compatibilidad con códigos QR de pago, que son generados por el sistema y leídos por dispositivos móviles del cliente mediante la cámara nativa. No se requieren impresoras de tickets ni escáneres de código de barras en la versión actual del sistema.
3.3.3 Interfaces de Software
El sistema interactúa con los siguientes servicios externos mediante sus APIs oficiales:

Clerk (autenticación): Gestión de sesiones, OAuth con Google, control de roles (user / admin) y verificación de correo electrónico. Stripe (pagos): Procesamiento de pagos con tarjeta, creación de Payment Intents y webhooks para confirmación de transacciones. Cloudinary (media): Almacenamiento, transformación y entrega optimizada de imágenes de productos. Inngest (jobs): Orquestación de tareas en segundo plano como envío de correos y actualización de estados. Nodemailer (correo): Envío de notificaciones de pedidos y facturas por correo electrónico. Google Maps API: Mapa interactivo para visualización de la ubicación de la cafetería.
3.3.4 Interfaces de Comunicación
Toda la comunicación entre el frontend (React), la aplicación móvil (React Native) y el backend (Node.js + Express 5) se realiza mediante API REST sobre HTTPS, con autenticación basada en JWT emitidos por Clerk.
El backend expone endpoints documentados bajo rutas versionadas (/api/v1/). Los webhooks de Stripe y Clerk se reciben en rutas protegidas con firma criptográfica. Las notificaciones de estado de pedido se envían por correo electrónico mediante Nodemailer a través de Inngest, garantizando entrega asíncrona y reintentos automáticos en caso de fallo.

4. APÉNDICES
4.1 Glosario
El presente glosario define los términos técnicos, funcionales y normativos empleados a lo largo de este documento, contextualizados al stack tecnológico y al alcance real del Sistema de Comercio Electrónico Don Palito Junior implementado en marzo de 2026.
React 18 + Vite 5: Stack frontend de la tienda web. React gestiona la interfaz de usuario mediante componentes reutilizables; Vite es el bundler y servidor de desarrollo con tiempos de arranque notablemente rápidos frente a alternativas como Create React App.
React Native + Expo SDK 52: Stack de la aplicación móvil. Permite desarrollar apps nativas para Android e iOS desde una base de código JavaScript compartida, aprovechando las APIs nativas del dispositivo.
Node.js + Express 5: Entorno de ejecución y framework del servidor backend. Node.js ejecuta JavaScript en el servidor; Express 5 gestiona las rutas HTTP, el middleware de autorización y la lógica de la API REST.
MongoDB Atlas: Base de datos NoSQL en la nube utilizada para almacenar productos, pedidos, usuarios, categorías, cupones y calificaciones. Ofrece escalabilidad automática, backups gestionados y acceso global con baja latencia.
Clerk: Servicio de autenticación e identidad. Gestiona el registro, inicio de sesión (incluido OAuth con Google), verificación de correo electrónico, sesiones JWT y control de roles (user / admin) sin necesidad de implementación propia.
Stripe: Pasarela de pagos utilizada para procesar pagos con tarjeta de crédito/débito. Gestiona Payment Intents y emite webhooks al backend para confirmar o rechazar transacciones de forma automática.
Cloudinary: Servicio de gestión de imágenes en la nube. Almacena, transforma y entrega las imágenes de productos con optimización automática de formato, resolución y carga diferida (lazy loading).
Inngest: Plataforma de orquestación de tareas en segundo plano (background jobs). Se utiliza para disparar envíos de correo, actualización de estados de pedido y otras operaciones asíncronas con reintentos automáticos ante fallos.
Nodemailer: Biblioteca Node.js para el envío de correos electrónicos transaccionales. Utilizada junto a Inngest para notificar a los clientes sobre el estado de sus pedidos y enviarles la factura de compra.
Tailwind CSS + DaisyUI: Herramientas de estilizado del frontend web. Tailwind CSS provee clases utilitarias de bajo nivel; DaisyUI agrega componentes preconstruidos (botones, modales, formularios, tarjetas) con soporte para temas de color.
Git / GitHub: Sistema de control de versiones y plataforma de alojamiento de repositorios. El equipo coordina el código del frontend, backend y app móvil mediante ramas y pull requests en repositorios compartidos.
JWT (JSON Web Token): Estándar abierto (RFC 7519) para transmitir información entre partes de forma segura. El backend verifica los JWT emitidos por Clerk en cada solicitud a la API para autenticar al usuario y determinar su rol.
API REST: Estilo arquitectónico para servicios web. El backend expone endpoints bajo rutas versionadas (/api/v1/) que el frontend y la app móvil consumen mediante peticiones HTTP estándar (GET, POST, PUT, DELETE).
Webhook: Notificación HTTP enviada automáticamente por un servicio externo (Stripe, Clerk) al backend cuando ocurre un evento relevante, como la confirmación de un pago o la creación de un usuario nuevo.
Catálogo de Productos: Sección de la tienda web y la app móvil que muestra los productos disponibles de la cafetería con imagen, nombre, descripción, precio y disponibilidad. Permite filtrado por categoría y búsqueda por nombre.
Código QR de Pago: Método de pago que permite a los clientes realizar transferencias bancarias escaneando un código QR desde su dispositivo móvil. El comprobante se envía al administrador vía WhatsApp para su validación manual.
Panel de Administración: Interfaz web exclusiva para administradores que permite gestionar productos, categorías, pedidos, cupones, facturas, calificaciones y estadísticas de ventas del negocio.
Cupón de Descuento: Código alfanumérico creado por el administrador que los clientes pueden aplicar durante el proceso de compra para obtener un descuento sobre el total del pedido.
Estado del Pedido: Etapa en el ciclo de vida de una orden: Pendiente → Pagado / Rechazado → Entregado. El administrador actualiza el estado manualmente desde el panel; el sistema notifica al cliente por correo en cada cambio.
Calificación de Producto: Evaluación numérica de 1 a 5 estrellas que un cliente puede dejar sobre un producto después de haberlo comprado. El administrador puede moderar y eliminar calificaciones desde el panel.
Factura: Documento electrónico generado automáticamente por el sistema cuando se confirma el pago de un pedido. Se envía al correo del cliente y queda disponible para descarga desde el panel de administración.
Personalización de Empaques: Funcionalidad que permite al cliente incluir una nota especial al momento de realizar su pedido, solicitando un diseño o mensaje personalizado en el empaque del producto.
IEEE 830-1998: Estándar del Institute of Electrical and Electronics Engineers para la documentación de Especificaciones de Requisitos de Software (ERS). Define la estructura mínima, el contenido y las características de calidad que debe tener un documento de requisitos.
WCAG 2.1: Web Content Accessibility Guidelines. Directrices del W3C para mejorar la accesibilidad del contenido web. El sistema aspira a cumplir el nivel AA en la interfaz de la tienda web, garantizando acceso a personas con diversidad funcional.
Ley 1581 de 2012: Ley colombiana de protección de datos personales (Habeas Data). Establece las condiciones para el tratamiento de datos de ciudadanos colombianos. El sistema la cumple mediante políticas de privacidad, consentimiento explícito y protección de los datos de clientes.
SENA: Servicio Nacional de Aprendizaje. Entidad pública colombiana de formación profesional. El presente proyecto es el trabajo de grado del programa Tecnología en Análisis y Desarrollo de Software del Centro del Diseño y Manufactura del Cuero (CDMC), Regional Antioquia.
Disponibilidad: Porcentaje de tiempo en que el sistema está operativo y accesible. El sistema debe estar disponible 24/7 con tiempo de inactividad mínimo, respaldado por la infraestructura de MongoDB Atlas y los servicios cloud utilizados.
Escalabilidad: Capacidad del sistema para manejar un aumento en el tráfico de usuarios sin afectar el rendimiento. La arquitectura basada en servicios cloud (MongoDB Atlas, Cloudinary, Clerk) facilita el escalado horizontal.
Responsividad: Propiedad de la interfaz de usuario que le permite adaptarse correctamente a distintos tamaños de pantalla: móviles, tablets y escritorio. Implementada con Tailwind CSS en la tienda web y con los componentes nativos de Expo en la app móvil.
Seguridad de Datos: Conjunto de medidas que protegen la información personal y transaccional de los usuarios. Se implementa mediante HTTPS, JWT firmados por Clerk, webhooks con firma criptográfica (Stripe) y cumplimiento de la Ley 1581 de 2012.
Tiempo de Carga: Tiempo máximo permitido para que el sistema responda a una solicitud del usuario. El estándar definido en los RNF establece menos de 3 segundos en conexión estándar, apoyado por la optimización de imágenes de Cloudinary y la caché del frontend.
Mantenibilidad: Facilidad para modificar, corregir y actualizar el sistema. Se garantiza mediante código documentado, separación de responsabilidades entre frontend, backend y app móvil, y uso de repositorios Git con historial de cambios.
Backup de Datos: Copias de seguridad automáticas de la base de datos realizadas por MongoDB Atlas de forma periódica. Permiten recuperar el sistema ante fallos graves sin pérdida significativa de información.
Control de Acceso: Mecanismo que restringe las funciones disponibles según el rol del usuario autenticado. Los roles definidos son user (cliente) y admin (administrador), gestionados por Clerk y verificados en cada endpoint del backend.
