Sistema de Comercio Electrónico – Don Palito Junior
Versión 2026

1. Introducción
1.1 Propósito del documento
El presente documento tiene como propósito definir de manera formal y estructurada los requisitos funcionales y no funcionales del sistema de comercio electrónico Don Palito Junior. Este documento sirve como referencia oficial para validación con el cliente, desarrollo, mantenimiento, evaluación académica y control de alcance.
1.2 Alcance del sistema
El sistema Don Palito Junior es una solución digital de comercio electrónico orientada a la gestión de ventas, inventario y atención al cliente para una cafetería. Permite registro y autenticación de usuarios, gestión de productos, carrito de compras, proceso de checkout, generación de pedidos, administración de inventario, gestión de promociones, facturación, notificaciones por correo electrónico y generación de reportes.
El sistema está compuesto por una aplicación web, una aplicación móvil desarrollada en React Native y un panel administrativo. El backend oficial está desarrollado en Node.js con base de datos MongoDB.
1.3 Definiciones y Acrónimos
ERS: Especificación de Requisitos de Software.
RF: Requisito Funcional.
RNF: Requisito No Funcional.
API: Interfaz de Programación de Aplicaciones.
Backend: Lógica del servidor y acceso a base de datos.
Frontend: Interfaz visual del usuario.
Checkout: Proceso de finalización de compra.
2. Descripción General del Sistema
2.1 Perspectiva del Producto
El sistema sigue una arquitectura cliente-servidor basada en una API REST. Las aplicaciones web y móvil consumen la misma API, garantizando coherencia funcional entre plataformas. Actualmente el sistema se encuentra desplegado en entorno local, con proyección de migración a infraestructura en la nube.
2.2 Funciones Generales del Sistema
Funciones para Clientes:
- Registro, autenticación y gestión de perfil.
- Visualización de catálogo y productos.
- Carrito de compras y aplicación de descuentos.
- Proceso de checkout con generación de pedido.
- Consulta de historial de pedidos.
- Calificación y reseñas de productos.
Funciones para Administradores:
- Gestión de cuentas de usuario.
- Gestión de inventario y categorías.
- Gestión de promociones.
- Gestión de pedidos y estados.
- Generación de reportes y estadísticas.
- Exportación de datos.

5. Arquitectura del Sistema
5.1 Arquitectura General
El sistema adopta una arquitectura cliente-servidor basada en una API REST. El backend está desarrollado en Node.js y utiliza MongoDB como sistema de base de datos NoSQL. Las aplicaciones cliente (Web y Mobile), así como el panel de administración consumen la misma API, garantizando coherencia funcional.
5.2 Componentes del Sistema
Backend: Implementado en Node.js, gestiona la lógica de negocio, autenticación, control de roles, procesamiento de pedidos, gestión de inventario y generación de reportes.
Base de Datos: MongoDB almacena información de usuarios, productos, pedidos, facturas, promociones, categorías y registros de auditoría, entre otros.
Aplicación Web: Interfaz accesible desde navegadores modernos para clientes y administradores.
Aplicación Mobile: Desarrollada en React Native, ofrece las mismas funcionalidades que la versión web orientada a clientes.
Panel Administrativo: Interfaz especializada para la gestión interna del negocio.
5.3 Seguridad Arquitectónica
El sistema implementa autenticación basada en credenciales, control de sesiones y validación de roles. Se aplican mecanismos de protección de rutas y validaciones en el backend para prevenir accesos no autorizados.
6. Reglas de Negocio
RN-01: Un usuario debe estar autenticado para realizar compras.
RN-02: Solo los administradores pueden modificar inventario y promociones.
RN-03: Un pedido solo puede cambiar a estado 'Pagado' tras validación manual del administrador.
RN-04: Un producto no puede venderse si su inventario es igual a cero.
RN-05: Un cliente solo puede calificar productos que haya comprado.
RN-06: Las promociones deben tener fecha de vigencia definida.
RN-07: Las facturas solo se generan cuando el pago es confirmado.
7. Modelo de Datos - Descripción Conceptual
El modelo de datos está basado en una estructura NoSQL utilizando MongoDB. Las principales entidades del sistema incluyen:
- Usuario (Cliente / Administrador).
- Producto.
- Categoría.
- Promoción.
- Pedido.
- Factura.
- Reseña.
- Registro de Auditoría.
Las relaciones se gestionan mediante referencias entre documentos y estructuras embebidas según la necesidad del módulo correspondiente.
8. Consideraciones de Despliegue
Actualmente el sistema se encuentra desplegado en entorno local para pruebas y desarrollo.
Se proyecta migración a infraestructura en la nube utilizando servicios como Hostinger o Digital Ocean, lo cual permitirá mayor disponibilidad y escalabilidad.
El despliegue futuro deberá contemplar configuración de servidor, base de datos en la nube, certificados de seguridad (HTTPS) y configuración de dominios.
